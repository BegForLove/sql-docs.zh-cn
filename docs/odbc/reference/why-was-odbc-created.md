# <a name="why-was-odbc-created"></a>为何创建 ODBC？
从历史上看，公司使用单个 DBMS。 所有数据库访问都通过该系统的前端或通过编写的应用程序来完成，这些应用程序完全用于该系统。 然而，随着计算机的使用越来越多，计算机硬件和软件越来越多，公司开始购买不同的DBMS。 原因很多：人们买了最便宜的，最快的，他们已经知道的，市场上最新的，什么最适合一个单一的应用程序。 其他原因包括重组和合并，以前只有一个 DBMS 的部门现在有几个。  
  
 随着个人电脑的出现，这个问题变得更加复杂。 这些计算机带来了一系列用于查询、分析和显示数据的工具，以及许多价格低廉、易于使用的数据库。 从那时起，单个公司的数据通常分散在无数的桌面、服务器和微型计算机中，存储在各种不兼容的数据库中，并且由大量不同的工具访问，其中很少可以获取所有数据。  
  
 最后的挑战来自客户端/服务器计算的出现，它寻求最有效地利用计算机资源。 廉价的个人计算机（客户端）位于桌面上，同时提供数据的图形前端和一些廉价的工具，如电子表格、图表程序和报表生成器。 微型计算机和大型计算机（服务器）承载 DBMS，在那里他们可以使用其计算能力和中心位置提供快速、协调的数据访问。 那么前端软件是如何连接到后端数据库的呢？  
  
 独立软件供应商 （ISV） 也面临着类似的问题。 为小型计算机和大型机编写数据库软件的供应商通常被迫为每个 DBMS 编写一个版本的应用程序，或为他们想要访问的每个 DBMS 编写特定于 DBMS 的代码。 为个人计算机编写软件的供应商必须为他们想要使用的每个不同的 DBMS 编写数据访问例程。 这通常意味着大量的资源用于编写和维护数据访问例程而不是应用程序，而应用程序通常不是以质量出售，而是销售它们是否可以访问给定 DBMS 中的数据。  
  
 这两组开发人员都需要一种访问不同 DBMS 中数据的方法。 大型机和微型计算机组需要一种方法来合并单个应用程序中来自不同 DBMS 的数据，而个人计算机组需要此功能以及编写独立于任何一个 DBMS 的单个应用程序的方法。 简而言之，两个组都需要一种可互操作的方式来访问数据;他们需要开放的数据库连接。
